## Highlights

OpenAGt v1.21.0 ships the domain-specialized agentic helper runtime: a
critical-review pipeline (MPACR), a three-layer memory architecture (3LMA) that
accumulates domain knowledge across sessions, and a dynamic expert / prompt
template registry (DXR + PTS) that lets users extend the coordinator without
touching source code.

This release is the largest behavioral expansion since v1.16. New behavioral
surfaces remain opt-in or manually invoked: existing v1.20 configurations keep
their default plan shape unless the new feature flags or CLI commands are used.
The builtin coordinator roles remain schema-compatible; user experts attach via
an additive registry.

Test coverage for the affected subsystems is **239 passing tests** with zero
typecheck errors across `packages/openagt`.

## Key Features

### MPACR - Multi-Perspective Adversarial Critical Review (Stream A)

A four-stage structured-debate replacement for the legacy single-pass
`reviseNode`. Behind `EffortProfile.mpacr_enabled`. When enabled, every
plan-/output-/handoff-/reducer-/verifier-revise gate that has a concrete target
artifact is expanded into a debate graph:

1. **Steel-man** - restate the artifact's strongest reading
2. **Red-team (parallel)** - K critics attack from preset perspectives
   (`factuality`, `coherence`, `risk`, `domain_expertise`, `user_value`)
3. **Defense** - owner rebuts or concedes each critique
4. **Synthesis** - emit a `CriticalReviewVerdict` with constructive feedback
5. **Calibration** - record posterior + Brier feedback for long-term tuning

- `CriticalReviewVerdict` schema gains `evidence_for`, `evidence_against`,
  `priors`, `posterior`, and `brier_score`. Reviewers must populate evidence on
  both sides; symmetric epistemology is enforced by `mpacr-validation`.
- Coordinator revise output handling calls `validateCritique`. Critic outputs
  that still fail evidence validation after retry move into the low-confidence /
  ask-user path instead of silently passing.
- MPACR critic task metadata carries `mpacr_per_critic_timeout_ms` (default
  180 s). The coordinator enforces that timeout at runtime.
- MPACR critic executor absence, thrown errors, and timeouts are converted into
  completed synthetic `verdict: "skipped"` outputs. Generic task dependency
  semantics are unchanged.
- Synthesis tolerates skipped critics up to the configured quorum. Default
  quorum is `max(1, ceil(K * 0.6))` (K=3 -> 2, K=5 -> 3, K=6 -> 4).
- Degraded MPACR graph planning is conservative: it is selected only when the
  provided budget options explicitly mark the run as budget-limited or cannot
  satisfy the full MPACR node count.
- Brier-score calibration rows persist per `(expert_id, workflow)` pair to the
  new `calibration_record` table. `Calibration.recommendPriorAdjustment`
  returns a recommended shift in `[-0.1, +0.1]` once the sample exceeds
  `MIN_CALIBRATION_SAMPLES` (20).

### 3LMA - Three-Layer Memory Architecture (Stream B)

Adds **semantic** (cross-session domain facts) and **procedural** (successful
task recipes) layers atop the existing episodic memory. All three layers share
the existing `personal_memory_note` table; no new storage backend is required.

- `MemoryScope` enum extended with `"semantic"` and `"procedural"`.
- `ThreeLayerMemory.recordSemanticFact / recordProceduralRecipe /
  searchSemantic / searchProcedural / detectDomain` are exposed as an Effect
  Service.
- **Memory Consolidator** background process: encode -> pattern-extract ->
  replay -> decay. Triggered manually via `openagt mem consolidate`; scheduler
  integration remains roadmap.
  - Pattern extraction: `(domain, subject, predicate, object)` triples
    repeating at least `min_pattern_occurrences` (default 3) become semantic
    facts with `confidence = min(0.95, n / 10)`.
  - Replay: when a semantic key already exists, the consolidator updates the
    existing semantic note's `source_note_ids`, `rehearsal_count`, confidence,
    importance, and `time_updated` instead of creating a duplicate fact.
  - Source episodic notes are marked `metadata.consolidated = true` after both
    replay and new semantic encoding paths.
  - Decay: `keep` writes refreshed importance/metadata, `demote` moves the note
    out of semantic retrieval scope, and `delete` is a soft delete that also
    moves the note out of semantic scope. No memory data is hard-deleted.
- **Cross-process advisory locks** (`_advisory_lock` table, 5-minute TTL,
  PID + machine-ID guard) prevent multiple `openagt` processes from
  double-consolidating.
- **WAL checkpoint protection**: consolidator calls `walCheckpointTruncate`
  every configured write batch to bound WAL growth.
- `enrichPlanMemory` Effect post-processor injects domain-relevant semantic
  facts and procedural recipes into `plan.memory_context.note_ids` at plan
  creation time. It is best-effort and falls back to the bare plan on backend
  hiccups.
- `taskSignatureFor({goal, workflow})` produces a stable lookup key
  (`${workflow}:${domain}:${sorted_top3_keywords}`) so a recipe recorded for one
  task is reused for the next compatible task.

### DXR + PTS - Dynamic Expert Registry & Prompt Template System (Stream C)

User-extensible expert roles plus centralized prompt templates with A/B
testing. The closed builtin `CoordinatorNodeRole` contract stays compatible:
user experts attach by inheriting from a builtin role.

- `.opencode/experts/*.md` loader mirrors the agent/command frontmatter pattern.
  Loaded into Config under the `expert` key during dir-walk.
- `ExpertRegistry` Effect Service lazy-initializes from Config on first query
  and falls back to builtins when Config is unavailable in tests.
- `applyExpertOverride(node, entry)` replaces prompt, expert_id, and
  memory_namespace while preserving `role` and `output_schema`, so EffortProfile
  dispatch and verdict parsing keep working transparently.
- `applyUserExpertsToPlan(plan)` applies user experts only when the inherited
  parent role matches and the expert's workflow/domain metadata is compatible
  with the plan node. This prevents review-only or domain-specific experts from
  bleeding into unrelated workflows.
- **Prompt Templates**: 7 builtin templates extracted from
  `coordinator/*Node()` factories to `coordinator/prompts/<role>/<variant>.md`
  with byte-equal snapshot tests guarding against drift:
  `reviser/{default,no-target}`, `reviewer/checkpoint`, `planner/default`,
  `verifier/shard`, and `reducer/{default,project-deep-dive}`.
- **Prompt telemetry**: `prompt_outcome` records template role, variant,
  task_id, expert_id, success, and duration. The picker uses stored outcomes for
  Thompson sampling-lite with a 5% exploration floor.

### Database & Migration Safety Net

- **`_schema_version`** audit table records every applied migration with
  SHA-256 checksum. Backfilled on first upgrade.
- `openagt db status` lists applied migrations and runs `PRAGMA integrity_check`
  by default. Pass `--no-integrity` to skip the check for large databases.
- **`_advisory_lock`** supports cross-process coordination for the consolidator.
- **`calibration_record`** stores Brier-score history.
- **`prompt_outcome`** stores per-variant prompt telemetry.
- All four new migrations are append-only and idempotent.

### CLI Surface

| Command | Purpose |
|---|---|
| `openagt db status [--no-integrity]` | List applied migrations and run integrity check by default |
| `openagt experts list [--source builtin\|user] [--domain X]` | List registered expert roles |
| `openagt experts show <role>` | Show full expert definition |
| `openagt cal show [expert_id] [--since unix-ms]` | Brier score and recommended prior shift per expert |
| `openagt prompt stats [role] [--days 30]` | Per-variant outcome stats from prompt_outcome telemetry |
| `openagt mem consolidate` | Manually run a memory consolidation pass |

All commands support `--format json` for ops automation.

## Migration & Compatibility

- **Zero breaking changes.** Existing configs, sessions, and databases load
  unchanged. New schema fields all have safe defaults.
- New role enum entries (`steel-manner`, `red-team-critic`, `defender`,
  `synth-reviser`, `calibrator`) are appended to `CoordinatorNodeRole`; the
  builtin role contract is preserved.
- `EffortProfile.mpacr_enabled` defaults to `false`, so v1.20 plan shapes remain
  byte-identical to v1.21 plan shapes when MPACR is off.
- `MemoryScope` and `CoordinatorPlan.memory_context.scopes` now accept
  `"semantic"` and `"procedural"`. Existing notes load unchanged; new scopes are
  only used by ThreeLayerMemory writes and enrichment.
- Coordinator test fixtures that construct their own Layer stack must provide
  `ThreeLayerMemory` and `ExpertRegistry` alongside the coordinator services.

## Hazard Register Coverage

The 15 hazards documented in the v1.21 implementation plan
(`.claude/plans/potencial-harzard-replicated-ullman.md`) are tracked below.
v1.21 only claims closure for rows with runtime wiring and tests; partial rows
remain roadmap or operational hardening work.

| # | Hazard | Status | Runtime / test coverage |
|---|---|---|---|
| H1 | Single critic fail blocks synthesis | Covered | MPACR critics that lack an executor, throw, or time out complete with synthetic `skipped` verdicts; downstream dependencies unblock. |
| H2 | K critics share parent wallclock | Covered | `mpacr_per_critic_timeout_ms` is carried on critic task metadata and enforced by the coordinator. |
| H3 | `evidence_against` retry cascade | Covered | `validateCritique` is called from coordinator revise output handling; repeated validation failure enters ask-user / low-confidence handling. |
| H4 | Models fake evidence to pass schema | Partial | Calibration rows and prompt-outcome rows are produced; calibration-informed expert weighting remains a tuning follow-up. |
| H5 | DB migration leaves half-state | Covered | `_schema_version` plus `db status` default integrity check; CLI smoke covers the default. |
| H6 | FTS5 trigger drift | Roadmap | Semantic search works through existing personal-memory search; explicit trigger-drift regression remains pending. |
| H7 | Pattern extraction creates duplicate or poisoned facts | Covered | Confidence cap plus replay dedupe tests; existing semantic keys are updated instead of duplicated. |
| H8 | WAL bloat during consolidation | Implemented | Consolidator checkpoints every configured write batch; config sanity is tested. |
| H9 | Multi-process double consolidation | Implemented | `_advisory_lock` TTL path is implemented; cross-process stress coverage remains pending. |
| H10 | User expert vs parent schema conflict | Covered | Overrides preserve parent role/output schema and respect workflow/domain scope. |
| H11 | Prompt extraction drift | Covered | Byte-equal prompt snapshot tests guard extracted templates. |
| H12 | A/B variant cold-start starvation | Covered | Thompson sampling with 5% exploration and cold-variant tests. |
| H13 | Outcome attribution wrong | Covered | `prompt_outcome` records task/template role, variant, task_id, expert_id, success, and duration. |
| H14 | Bundled prompts lost in build | Implemented | Template loader supports bundled prompts and filesystem fallback. |
| H15 | PR misses stream regressions | Roadmap | Local verification is green; PR-blocking CI expansion remains release-process work. |
